# Вопрос 3 (Session 5: Datalog) — `reach`

Правила:

```
reach(X,Y) :- g(X,Y).
reach(X,Y) :- g(X,Z), reach(Z,Y).
```

Факты `g`:
`{(a,b),(b,c),(c,d),(d,e),(f,g),(g,h),(h,i),(b,i)}`

## 3(a) Почему semi-naive неэффективен для `reach(c,Y)`

Semi-naive recursion вычисляет `reach` “снизу вверх” по всей базе, а не только для начального узла `c`.

На первых шагах в `Δreach` попадают не только вершины, достижимые из `c`, но и достижимые “из других источников” из `g`.

Например:

1. `Δ¹` включает все прямые дуги (факты `g`), то есть стартовые узлы `a,b,c,d,f,g,h,b` уже присутствуют одновременно.
2. На следующем шаге рекурсия `g(X,Z) ⋈ reach(Z,Y)` продвигает достижимость дальше по дугам для всех `X`, которые встречаются в `g`, в том числе для `X=f,g,h` (компонента `f->g->h->i`), хотя запрос просит только достижимые из `c`.

Поэтому даже чтобы ответить на запрос `reach(c, Y)`, система всё равно накапливает intermediate-результаты для `X` из других частей графа.

## 3(b) Top-down (goal-driven) вычисление `reach(c,Y)`

Ищем `reach(c,Y)`:

1. По правилу базы: `reach(c,Y) :- g(c,Y)`.  
   Из фактов `g` имеем `g(c,d)`, значит `Y=d`.
2. По рекурсивному правилу: `reach(c,Y) :- g(c,Z), reach(Z,Y)`.  
   снова `g(c,Z)` даёт только `Z=d`, значит нужно `reach(d,Y)`.
3. Вычисляем `reach(d,Y)`:
   - база: `g(d,Y)` даёт `Y=e`
   - рекурсия для `d` требует `reach(e,...)`, но дуг из `e` в данных нет, значит продолжений нет.

Ответ:
`reach(c,Y) = {(c,d),(c,e)}`.

## 3(c) Magic sets для запроса `reach(c,Y)`

Добавленные правила:

```
m(c).
m(Z) :- m(X), g(X,Z).

reach(X,Y) :- m(X), g(X,Y).
reach(X,Y) :- m(X), g(X,Z), reach(Z,Y).
```

1) Вычисляем magic-предикат `m` (какие `X` нужны):

- `m(c)` (дано)
- из `m(c)` и `g(c,d)` получаем `m(d)`
- из `m(d)` и `g(d,e)` получаем `m(e)`

Итого: `m = {c,d,e}`.

2) Дальше `reach(X,Y)` строится только для `X` из `m`.

- `reach(c,d)` из `m(c), g(c,d)`
- `reach(d,e)` из `m(d), g(d,e)`
- затем по рекурсии получаем `reach(c,e)` из `m(c), g(c,d), reach(d,e)`

Ответ тот же, но вычислялось меньше `X`:
`reach(c,Y) = {(c,d),(c,e)}`.

## 3(d) Переписать запрос как `reach(c,e)`

В goal-driven / magic-sets подходе связываем второй аргумент константой `e`.

Эквивалентно можно завести специализированный предикат, например:

```
me(c).
me(Z) :- me(X), g(X,Z).

reachToE(X) :- me(X), g(X,e).
reachToE(X) :- me(X), g(X,Z), reachToE(Z).

ответ: reach(c,e) тогда и только тогда, когда reachToE(c) истинно.
```

Смысл: magic ограничивает множество кандидатов для первого аргумента теми `X`, которые достижимы от `c`, а `g(X,e)` оставляет только путь, который заканчивается в `e`.

## 3(e) Что если запрос `reach(X,e)`?

Нужно найти все вершины `X`, для которых существует путь `X ->* e`.

По данным фактам путь к `e` возможен только так:
- `d -> e` даёт `X=d`
- `c -> d -> e` даёт `X=c`
- `b -> c -> d -> e` даёт `X=b`
- `a -> b -> c -> d -> e` даёт `X=a`

Ответ:
`{X | reach(X,e)}` = `{a,b,c,d}` (а `reach(X,e)` истинно для пар `(a,e),(b,e),(c,e),(d,e)`).
